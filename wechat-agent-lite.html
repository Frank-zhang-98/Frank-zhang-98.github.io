wechat-agent-lite

这是一个面向微信公众号持续更新场景的轻量自动化系统。FastAPI 提供控制台和 API，APScheduler 定时触发 `health` / `main` 任务，在小规格服务器上完成抓取、筛选、写作、配图、草稿发布和日报回传。

主流程由 `RuntimeFacade` 驱动，运行态写入 SQLite：`runs`、`run_steps`、`llm_calls`、`source_health_states` 都能直接拿来排障、复盘和手动重跑。

项目概览

项目概览

项目把任务编排、状态持久化、抓取维护和发布接口放进同一个服务里，重点是持续运行而不是单次成功。

它解决什么问题

当抓取、模型调用、图像处理和发布动作散落在不同脚本里时，系统既难维护，也难定位问题。这里把它们收束进同一条执行链，并把配置、步骤结果和异常信息一起持久化。

它适合什么场景

适合需要稳定地产出公众号内容、但基础设施和维护预算都有限的场景，尤其适合 2C2G 一类小服务器部署，以及人工审核与自动流程并存的团队。

核心流程

核心流程能力

调度层只区分 `health` 和 `main` 两类任务，执行层再拆成明确的 step，并为每一步保留输入、状态和回退边界。

01
Health Check / Source Maintenance 启动前做健康检查，并持续记录来源状态；必要时尝试修复、降权、切换来源或停用。
02
Fetch / Dedup / Rank / Select 从多个来源抓取候选内容，去重后通过规则打分与模型重排来确定最终选题。
03
Write / Quality Check / Render 进入写作、事实补全、质量检查和渲染阶段，确保最终内容不只是生成出来，还能顺利进入发布链路。
04
WeChat Draft / Daily Report 将结果发布到公众号草稿，并通过日报或控制台输出运行情况，让系统具备操作和复盘价值。

技术栈

技术栈与结构

技术选型偏稳健与直接，优先服务于可靠运行、数据库可追踪性和低资源部署，而不是追求复杂堆栈。

服务与调度

使用 Python、FastAPI 和 APScheduler 作为入口骨架，`RuntimeFacade` 负责触发主流程、health run、manual run 和 step redo。

Python FastAPI APScheduler

数据与状态

通过 SQLAlchemy 组织 `config_entries`、`runs`、`run_steps`、`llm_calls` 和 `source_health_states`，把运行态沉淀成可查询结构。

SQLAlchemy Config Entries Runtime Records

抓取与页面能力

抓取侧结合 requests、feed 解析和 Playwright，并保留 Scrapling fallback 与按 host 限流策略，兼顾普通抓取、复杂页面和低资源机器约束。

Requests Scraping Playwright
设计重点

设计重点

这些点决定了它更像一个可长期维护的系统，而不是一次性的自动化脚本集合。

先做可靠性，再谈炫技 并发默认按 host 收到 1，Scrapling 走单独限流，优先避免打爆 2C2G 机器、反爬和 SQLite 写竞争。
让人工干预成为正式能力 关键节点保留人工判断空间，状态也更容易读懂。
把内容生产当作整条链路设计 难点在于把来源、生成和发布连成稳定闭环。
