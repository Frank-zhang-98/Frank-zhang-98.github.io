赛博丁真 QQ 群机器人

目标是把群聊里的资料沉淀为可检索的知识库，并在被 @ 时提供有依据、可追溯的回答。系统由“数据录入/维护”与“群聊响应”两条链路协作完成。

查看源码 返回项目区

项目定位

  • 面向 QQ 群的知识检索机器人，强调“可追溯、可复用”。
  • 把图文/表格/结构化资料统一纳入检索库，减少记忆负担。
  • 用流程化的 RAG 检索与对话记忆提升回答质量。

技术栈

核心技术来自代码与依赖配置，覆盖机器人框架、检索引擎、向量库、UI 与模型服务。

Python NoneBot2 OneBot v11 LangGraph Streamlit Qdrant BM25 (rank-bm25) FlashRank Jieba SQLite DeepSeek (OpenAI 兼容) Qwen-VL 图像解析 httpx / requests python-dotenv

设计方案（系统拆解）

整体方案遵循“内容入库 → 混合检索 → 提示词拼装 → 群聊应答”的闭环设计。

链路 A · 内容入库

Streamlit 录入/维护台

扫描目录或上传文件，支持图片结构化、文档预览编辑、增量更新与删除同步。

链路 B · 检索引擎

向量 + BM25 双检索

将分片写入 Qdrant，配套 BM25 索引与 FlashRank 重排，保证召回与精排。

链路 C · 群聊响应

LangGraph 流程编排

入口解析 → 触发判断 → 记忆加载 → RAG 检索 → LLM 生成 → 存储总结。

核心流程

  • 文本/图片/CSV/JSONL 统一解析为文档对象，写入 `all_docs.jsonl`。
  • 按 chunk_size + overlap 切分，产出 `chunks.jsonl` 并生成 embedding。
  • Qdrant 执行向量检索，BM25 负责关键词检索，二者候选合并后用 FlashRank 重排。
  • 群聊事件进入 LangGraph：判断是否 @ 机器人 → 取历史与摘要 → 检索 worldbook → 生成回复。
  • 消息写入 SQLite，按会话维度做 12 条一次的摘要更新。

数据与存储设计

结构化文件

  • `data/structured/all_docs.jsonl`：原始文档集合。
  • `data/structured/chunks.jsonl`：分片结果，便于排查。
  • `data/bm25/index.pkl`：BM25 索引。

会话记忆

  • `data/bot.sqlite`：记录群聊消息与摘要。
  • 按群 + 用户 + 角色维度生成 session key。
  • 摘要用于压缩长期记忆，避免提示词过长。

运行方式

运行前需要配置 `.env`，确保 Qdrant 与模型服务可用。

  • 启动数据录入工具：`streamlit run app_streamlit.py`
  • 启动 QQ 机器人：`python -m bot.main`
  • 本地对话测试：`python tools/chat_local.py`
  • OneBot v11：在 NapCat/LLOneBot 配置反向 WS。
